# 看板、库存、移库、转包和封存设计说明

更新时间：2026-06-30

本文说明 WMS 当前的箱级看板模型，以及移库、转包、第三方仓、封存与普通出库之间的区别。

## 一、总原则

系统里一个看板只代表一个真实箱子：

```text
一个看板 = 一个箱子 = 一个当前数量 = 一个当前库位
```

因此，一个看板不能同时表示“部分在自有仓、部分在第三方仓”。一旦一个箱子的数量被拆开到不同位置，就必须拆成两个看板。

## 二、看板不是入库单父对象

入库单二维码类似早期“父看板”的批量扫码能力，但它不是看板对象，不保存到 `kanban` 表：

```text
入库单二维码：WMS-INBOUND|入库单号
出库单二维码：WMS-OUTBOUND|出库单号
箱级看板二维码：WMS-KANBAN|看板号|条码
```

这样可以避免三类混乱：

- 看板生命周期和单据生命周期混在一起。
- 一个入库单下多个箱子被误认为父子看板。
- 转包、部分出库时父看板状态无法准确表达真实库存。

## 三、看板主状态

看板主状态只表达生命周期主事实，组合状态由数量字段和分配表推导。

| 状态 | 中文显示 | 含义 |
| --- | --- | --- |
| `CREATED` / `WAIT_SCAN` | 待入库 | 看板已生成，箱子未扫码入库 |
| `INBOUND` / `IN_STOCK` | 自有仓在库 | 箱子在自有仓库中 |
| `ALLOCATED` | 已分配待出库 | 看板有数量被出库单锁定 |
| `PARTIAL_OUTBOUND` | 部分出库 | 箱子部分数量已客户出库 |
| `OUTBOUND` | 已出库 | 箱子数量已被客户出库消耗完 |
| `THIRD_PARTY_STOCK` | 第三方在库 | 箱子在第三方仓库中 |
| `FROZEN` | 已封存 | 箱子不可出库、不可移库、不可转包 |
| `CONSUMED` | 已消耗 | 原箱因出库或拆分后数量归零 |
| `RETURNED` | 已退回 | 入库单退回或看板作废 |

不建议增加类似“部分转包且部分出库”这样的爆炸式状态。前端应从下面这些数量字段组合展示：

```text
当前数量
可用数量
出库锁定数量
转包/移库锁定数量
已客户出库数量
当前位置和仓库性质
```

## 四、关键数量字段

| 字段 | 含义 |
| --- | --- |
| `qty` | 当前箱子数量 |
| `available_qty` | 当前可操作数量 |
| `reserved_qty` / `reserved_outbound_qty` | 已被出库单锁定数量 |
| `reserved_transfer_qty` | 已被移库或转包单锁定数量 |
| `outbound_qty` | 已客户出库消耗数量 |
| `source_kanban_id` | 如果该看板由拆箱产生，记录来源看板 |
| `transfer_order_no` | 最近关联的移库或转包单据号 |

可用数量原则：

```text
available_qty = qty - 出库锁定数量 - 转包/移库锁定数量
```

同一个数量不能被出库和转包重复锁定。

## 五、仓库性质

库位必须有仓库性质：

| 值 | 中文显示 | 含义 |
| --- | --- | --- |
| `OWN` | 自有仓库 | 公司内部库存位置 |
| `THIRD_PARTY` | 第三方仓库 | 外协、转包或第三方暂存位置 |

看板当前所属仓库由 `location_id` 关联得到。看板信息页面需要展示：

```text
仓库名称
库区名称
具体库位
仓库性质
```

并支持按仓库、库区、仓库性质筛选。

## 六、普通出库、移库、转包的区别

| 操作 | 来源 | 目标 | 是否减少系统总库存 | 说明 |
| --- | --- | --- | --- | --- |
| 普通出库 | 自有仓 | 客户 | 是 | 库存离开系统 |
| 自有移库 | 自有仓 | 自有仓 | 否 | 只改变库位 |
| 转包 | 自有仓 | 第三方仓 | 否 | 库存仍在系统，只是归属第三方仓 |
| 转包返还 | 第三方仓 | 自有仓 | 否 | 第三方仓库存返回自有仓 |
| 封存 | 当前库位 | 当前库位 | 否 | 只改变可用性 |

转包不能实现成普通出库。普通出库会减少系统总库存，转包只改变仓库性质和库位。

## 七、整箱转包

如果转包数量等于看板当前可转移数量，直接移动原看板位置。

示例：

```text
KB001 在总仓 A 区，数量 100，仓库性质 OWN
转包 100 到第三方仓 外协区
```

执行后：

```text
KB001 在第三方仓 外协区，数量 100，仓库性质 THIRD_PARTY
状态：THIRD_PARTY_STOCK
系统总库存不变
```

## 八、部分转包和第三方箱级看板

如果转包数量小于看板当前可转移数量，必须拆出新看板。

示例：

```text
KB001 在自有仓，数量 100
转包 30 到第三方仓
```

执行后：

```text
KB001 在自有仓，数量 70
KB002 在第三方仓，数量 30，source_kanban_id = KB001.id
```

新增的 `KB002` 是第三方箱级看板。它不是采购入库产生的新箱，而是由原自有仓箱拆分迁移产生的库存位置看板。

这样做会让系统里的看板数量增加，这是正确结果。因为现实中库存已经被拆成两个位置，系统必须用两个看板表达。

## 九、移库、转包和封存页面

当前前端功能拆分：

| 页面 | 路径 | 说明 |
| --- | --- | --- |
| 库存迁移/转包 | `frontend/src/components/pages/operations/InventoryOpsPage.vue` | 统一处理自有移库、转包到第三方、第三方返还 |
| 移库浮窗 | `frontend/src/components/pages/operations/inventory-ops-modals/TransferModal.vue` | 自有仓之间迁移 |
| 转包浮窗 | `frontend/src/components/pages/operations/inventory-ops-modals/RepackTransferModal.vue` | 转包到第三方或第三方返还 |
| 封存浮窗 | `frontend/src/components/pages/operations/inventory-ops-modals/FreezeModal.vue` | 对看板执行封存 |

旧的“转包结余”不再作为独立业务存在。结余由看板当前数量、拆分看板和库存流水自然表达。

## 十、看板信息页面

看板信息页面应以分页方式查询，避免一次性拉取所有看板。

接口：

```text
GET /api/kanbans/page
```

建议筛选项：

```text

库区名称
仓库性质
```

## 十一、库存看板统计

库存看板不能只显示总库存，至少应区分：

```text
总库存
自有仓库存
第三方库存
可出库库存
锁定库存
封存库存
```

推荐计算：

```text
总库存 = 自有仓库存 + 第三方库存
可出库库存 = 自有仓、未封存、未锁定的 available_qty
第三方库存 = 当前库位仓库性质为 THIRD_PARTY 的看板数量
```

库存明细、看板明细、库存流水都应在点击后分页拉取，避免进入页面时同时加载大量数据。

## 十二、验证标准

最小验证链路：

```text
1. 创建入库单。
2. 扫码入库，生成自有仓库存。
3. 创建普通出库单，确认只锁定库存，不立即扣减。
4. 扫码出库，确认总库存减少。
5. 创建自有移库，确认总库存不变，只改变库位。
6. 创建整箱转包，确认总库存不变，看板仓库性质变为 THIRD_PARTY。
7. 创建部分转包，确认原看板数量减少，并生成第三方箱级看板。
8. 封存看板，确认不可再出库、移库或转包。
```

如果转包后系统总库存减少，说明转包被错误实现成了普通出库。
